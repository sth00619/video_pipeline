# 태호님 피드백 대응 — 원인 진단 및 수정 지시 (2026-07-10)

> 4개 피드백 항목 전부 실제 저장소 코드(main 브랜치, 2026-07-10 클론 기준)를 열어 원인을 특정했습니다. "아마 이래서 그럴 것"이 아니라 해당 라인을 직접 인용합니다. 링크로 주신 참고 영상 3개는 제가 영상 자체를 재생·시청할 수 없어 내용은 확인하지 못했고, 이 문서는 코드 실사 결과만으로 작성했습니다 — 영상들이 구체적으로 보여주는 디테일이 있다면 추가로 알려주시면 반영하겠습니다.

## 1. "배경이 계속 비슷하다" — 원인 특정 완료

**원인**: `script_worker.py`의 `_generate_visual_prompt()` 함수(407~456행)를 보면, 배경 프롬프트는 `_VISUAL_KEYWORD_MAP`이라는 15개 카테고리 키워드 목록과 씬 텍스트를 매칭해서 결정됩니다. 그런데:

1. 어느 카테고리에도 안 걸리면 무조건 이 기본값으로 떨어집니다 (444행):
   ```python
   bg_prompt = "professional financial news studio background, dark navy blue background (#0d1b2a), glowing data screen"
   ```
   태호님이 "뉴스 컨셉이라서 스튜디오에서 계속 나오는 것 같다"고 느끼신 게 정확히 맞습니다 — **코드에 실제로 "financial news studio background"라는 문자열이 기본값으로 박혀 있습니다.**

2. 하나의 영상이 예를 들어 "삼성전자 반도체"처럼 **한 주제로 20분을 채우는 구조**라서, 그 영상 안의 거의 모든 씬이 같은 카테고리(예: "반도체" 키워드 매칭 → `"semiconductor chip circuit board glowing neon..."`)로 걸립니다. 카테고리 자체는 15개나 있지만, **한 영상 안에서는 결국 1~2개 카테고리로 수렴**하니 그 영상 전체가 비슷한 배경만 반복하게 됩니다.

**수정 방향**:
- `_generate_visual_prompt()`가 카테고리 매칭 문구만 쓰지 말고, **해당 씬의 실제 문장에서 언급된 구체적 대상(회사명, 수치, 사건)을 프롬프트에 추가로 삽입**하도록 확장 — 같은 "반도체" 카테고리라도 "삼성전자 3나노 수율" 씬과 "SK하이닉스 HBM 수출" 씬은 시각적으로 달라야 함
- `images_worker.py` 상단 주석에 이미 "섹션 유형별로 다른 시각화(intro/background/data/scenario/action/conclusion)"라는 설계 의도가 적혀있는데, 이게 matplotlib 폴백 렌더러에만 반영되고 AI 이미지 프롬프트 쪽(`_generate_visual_prompt`)에는 반영이 안 되어 있음 — 이 section 정보를 프롬프트 생성에도 연결
- 기본값(444행) 하나로 고정하지 말고, 최소 3~4종 배경 후보를 순환(rotation)시켜 매칭 실패 시에도 다양성 확보

## 2. "고정 캐릭터가 씬마다 다르게 느껴진다" — 원인 2가지 특정

지난번 개발 계획서(Phase B)에서 지적한 "채널별 캐릭터 레퍼런스 이미지가 프론트엔드에 연결 안 되어 있다"는 문제와 **직접 연결됩니다.** 추가로 이번에 코드를 더 파보니 두 번째 원인도 발견했습니다.

**원인 A (이미 파악된 것 — 재확인)**: `image.py`에 캐릭터 레퍼런스 이미지를 첨부하는 코드(`_generate_gemini_api`)는 있지만, `channelId`가 UI에서 한 번도 설정되지 않으므로 실제로는 **텍스트 설명만으로 매번 새로 그리는 상태**입니다. 텍스트 설명이 아무리 구체적이어도(`script_worker.py:450-454`의 `char_prompt`는 실제로 잘 쓰여 있습니다), AI 이미지 모델이 "이 정확한 얼굴 비율/눈 크기"를 기억할 방법이 없어 생성할 때마다 미세하게 달라집니다 — 이게 씬마다 "다른 캐릭터처럼" 보이는 근본 원인입니다.

**원인 B (신규 발견)**: `images_worker.py` 57행을 보면, 스크립트 JSON에 `sections`나 `scenes` 키가 정상적으로 안 채워져 있을 때 쓰는 폴백 코드가 있는데, 여기 하드코딩된 프롬프트가:
```python
"prompt": f"A cute green banknote cartoon character with glasses and a headset, showing an expression matching Scene {idx + 1}, clean 2D vector style"
```
**"초록색 지폐 캐릭터, 안경과 헤드셋 착용"** — 확정된 "금색 코인, 슈트, 지시봉" 캐릭터와 완전히 다른 설명입니다. 이 폴백이 걸리는 씬에서는 아예 다른 캐릭터가 나올 수밖에 없습니다. 이 폴백이 정확히 언제 걸리는지(스크립트 파싱 실패 빈도)는 추가 로그 확인이 필요하지만, **일단 이 하드코딩된 문자열 자체가 잘못됐으므로 즉시 수정 대상**입니다.

**수정 방향**:
- (최우선, 근본 해결) Phase B — 채널 프로필 생성 UI + 캐릭터 이미지 업로드 기능을 실제로 연결해서 레퍼런스 이미지 고정 방식으로 전환
- (즉시 수정 가능) `images_worker.py:57`의 하드코딩된 "green banknote... glasses and headset" 프롬프트를 실제 확정 캐릭터 설명(`script_worker.py:450-454`와 동일한 문구)으로 교체
- 이 폴백이 왜 걸리는지(스크립트 워커가 `sections`를 못 채우는 케이스가 실제로 있는지) 로그로 먼저 확인 — 원인 B가 자주 발생하는 것이면 스크립트 워커의 JSON 파싱 안정성 자체도 함께 봐야 함

## 3. "자막싱크가 아직도 안 맞는다" — 원인 특정 완료

`tts_worker.py`를 보면 4단계 폴백 체인이 있습니다: ElevenLabs Forced Alignment → stable-ts → faster-whisper → 글자수 비례. 문제는 **stable-ts와 faster-whisper 두 경로 모두 결국 같은 "글자 수 비례 매핑" 로직을 재사용**한다는 것입니다 (`tts_worker.py:516`에 "기존 Whisper 매핑 로직 재사용"이라고 직접 주석까지 있습니다).

즉, Whisper가 오디오에서 단어별 정확한 시간을 잡아내는 것까지는 정밀한데, 그 결과를 원본 대본 문장에 다시 이어붙이는 방식이 **"글자 수가 전체의 몇 %인지"로 시간을 나눠 배분**하는 방식입니다 (`tts_worker.py:534-535`, `620`). 이 방식은:
- "2,780포인트", "1,380.5원" 같은 숫자 — 글자 수는 적지만 실제 발음 시간은 김
- "FOMC", "CPI" 같은 영문 약어 — 한국어 TTS가 "에프오엠씨"처럼 풀어 읽으면서 글자 수 대비 훨씬 오래 걸림

이 두 케이스에서 정확도가 무너집니다. 그리고 **이 채널의 콘텐츠 자체가 지수/환율/금리 수치와 영문 약어로 가득 차 있으니** — 하필 가장 취약한 케이스가 이 채널의 핵심 콘텐츠와 정확히 겹칩니다. "아직도 안 맞는다"는 게 우연이 아니라 구조적인 문제입니다.

**수정 방향**: v3 문서의 "쇼츠 반자동화" 섹션(7.2절)에 이미 "Whisper 퍼지매칭으로 실제 컷포인트 역산"이라는, 글자 수 비례가 아니라 **텍스트 유사도 기반 매칭** 기법이 언급되어 있고 이건 쇼츠 추출 기능에는 이미 쓰이는 것으로 보입니다. 이 기법(예: `difflib.SequenceMatcher` 등 텍스트 시퀀스 정렬)을 롱폼 자막 타임스탬프 매핑에도 그대로 이식하면, 숫자·약어 구간에서의 드리프트가 크게 줄어들 것으로 판단됩니다.

**속도 관련**: 코드에서 실제 값을 확인해보니 현재 `atempo=1.35`(35% 가속)로 적용되어 있습니다 (`tts_worker.py:120`) — 기존에 알려졌던 "1.5배"와 다릅니다. 말씀하신 "속도는 좋은데 1.05배 정도 느려도 괜찮다"는, 원래 대표님 피드백이었던 "10~20% 빠르게"에 가깝게 다시 낮추자는 뜻으로 해석했습니다. 다만 정확히 "1.05배"가 최종 목표치인지, 아니면 "지금보다 조금 낮은 선(예: 1.15~1.2배)"을 말씀하신 것인지 표현이 조금 애매해서 — **정확한 목표 배속 수치는 한 번 더 확인 부탁드립니다.** (자막 재정렬이 걸리는 작업이라 한 번에 정확히 맞추고 넘어가는 게 좋을 것 같습니다.)

## 4. TTS 목소리 — "지식한입과 비슷하게" 관련 중요 확인 사항

이 부분은 진행 방향을 정하기 전에 꼭 한 번 짚어야 할 것 같아 먼저 말씀드립니다.

**확인된 사실**: 지식한입은 **AI 음성(TTS)이 아니라 실제 사람의 목소리로 진행하는 채널**입니다 (제작자 본인과 친구 두 명이서 채널을 운영한다고 알려져 있고, 실제 사람이 직접 내레이션합니다). 스타일은 평소보다 빠르고 하이톤에 재미있게 전달하는 편으로 알려져 있습니다.

**왜 이게 중요한가**: "TTS를 지식한입 목소리와 비슷하게"를 **그 사람의 목소리를 그대로 복제(클로닝)하는 것**으로 진행하면, 실존하는 특정 인물(공인이 아닌 개인 크리에이터)의 목소리를 본인 동의 없이 복제하는 셈이 되어 초상권/음성권 관련 법적 리스크가 있고, ElevenLabs 등 보이스 클로닝 플랫폼 약관에서도 본인 동의 없는 실존 인물 클로닝을 금지하고 있습니다.

**권장 방향**: "그 사람 목소리 자체"가 아니라 **"빠른 템포 + 하이톤 + 친근하고 재미있는 전달 방식"이라는 스타일/에너지**를 목표로 잡고, ElevenLabs Voice Library에서 비슷한 톤·속도 특성을 가진 기성 보이스를 찾거나 커스텀 튜닝하는 방향을 제안드립니다. 이러면 법적 리스크 없이 "듣기에 비슷한 느낌"은 충분히 재현 가능합니다. (혹시 이미 사내에서 확보한 자체 목소리 학습 소스가 있다면 그 쪽으로 Professional Voice Cloning을 진행하는 게 가장 깨끗한 방법입니다.)

## 5. 신규 기능 — "참고 대본 업로드" 기능

요청하신 기능(조회수 잘 나온 기존 대본을 업로드해서 키워드/구성 참고) 자체는 합리적이고, 스크립트 생성 파이프라인(3-Round 팩트체크)에 자연스럽게 끼워 넣을 수 있는 위치가 있습니다: Round 1(핵심 사실 추출) 이전에 "참고 대본"을 별도 입력으로 받아 **"이 대본의 구성 순서, 훅 방식, 문단 길이 배분"만 참고 스타일로 활용**하도록 프롬프트에 지시하는 식입니다.

다만 한 가지는 짚고 넘어가야 합니다: **다른 채널의 대본을 그대로 복사해서 실제 대본으로 쓰는 것("복붙해서 진행")은 저작권 문제**가 될 수 있습니다 (조회수가 잘 나온 남의 대본을 문장 단위로 그대로 가져다 쓰면 표절/저작권 침해 소지). 그래서 기능을 만들 때 아래처럼 설계하는 것을 권장합니다:

- 업로드된 참고 대본은 **"구조·키워드·전개 방식을 학습하는 참고 자료"**로만 LLM에 전달하고, "이 문장을 그대로 쓰지 말고 동일한 주제/수치를 우리 데이터로 재작성하라"는 지시를 명시
- UI에도 "참고용으로만 사용되며, 대본은 우리 채널 데이터 기준으로 새로 작성됩니다"라고 명확히 안내
- 이렇게 하면 원래 의도("조회수 잘 나오는 대본 구조를 배우고 싶다")는 그대로 살리면서 저작권 리스크는 피할 수 있습니다
